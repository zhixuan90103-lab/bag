# 多关卡尺寸适配方案

日期：2026-07-16  
项目：Pack 装箱游戏 Three.js 版本  
目的：在不改变摄像机和下方摆放物品区的前提下，支持 2x2、4x4、6x6 等不同尺寸关卡。

## 1. 核心约束

当前项目明确采用两个固定约束：

1. **摄像机固定**
   - 不因为小关卡自动推进摄像机。
   - 不因为不同关卡改变 `target`、`screenX`、`screenY`。
   - 只允许在大关卡确实装不下时拉远，且不改变构图中心。

2. **下方待放物品区固定**
   - 待放区最多显示 3 个物品。
   - 待放物品的预览尺寸、间距、位置不随关卡变化。
   - 玩家在不同关卡中看到的“待放物品栏”保持稳定。

这两个约束的目标是保证移动端竖屏构图稳定，避免切关后箱子漂移、待放物品被裁切、操作区域变化。

## 2. 不采用的方案

### 2.1 不采用视觉箱体和逻辑网格分离

曾讨论过 `2x2` 逻辑关卡显示成 `4x4` 视觉箱体的做法，但最终不采用。

原因：

- 玩家会看到多出来的箱底空间，却无法放置物品。
- “明明有空间为什么不能放”会造成认知偏差。
- 装箱游戏中，视觉空间必须和逻辑空间一致。

结论：**箱内格子、纸箱尺寸、放入后的物品尺寸必须完全跟逻辑一致。**

### 2.2 不采用按关卡缩放相机

小关卡如果通过推进相机放大，会导致：

- 下方待放物品区被裁切。
- 箱子在画面中位置发生变化。
- 玩家每关的手感不一致。

结论：**小关卡不能靠相机放大。**

## 3. 采用的方案

采用“两套尺寸基准”：

1. **箱内真实格子尺寸：`grid.cell` / `box.cellSize`**
   - 来自关卡数据。
   - 决定纸箱尺寸、箱底格子尺寸、放入后物品尺寸、层高、碰撞、吸附和体素占用。
   - 例如第一关 `2x2x1` 使用 `cellSize: 1.15`，让真实箱子和真实格子都更大。

2. **下方预览格子尺寸：`TRAY_CELL_SIZE = 0.78`**
   - 固定不变。
   - 只用于创建物品 mesh 的基准尺寸和待放区显示。
   - 不参与箱内逻辑判断。

物品从下方被拖入箱内时，通过缩放完成尺寸转换：

```js
boardScale = grid.cell / TRAY_CELL_SIZE
```

因此：

- 下方待放物品永远看起来统一。
- 箱内物品永远和当前关卡格子一致。
- 小关卡可以通过更大的 `cellSize` 让箱子变大。
- 不需要动摄像机。

## 4. 当前实现要点

### 4.1 固定预览尺寸

```js
const TRAY_CELL_SIZE = 0.78;
let itemCellSize = TRAY_CELL_SIZE;
let blockHeight = itemCellSize;
```

`createItemMesh()` 使用 `itemCellSize` 创建原始物品 mesh。  
由于 `itemCellSize` 固定为 `TRAY_CELL_SIZE`，所以所有待放区物品预览尺寸稳定。

### 4.2 关卡真实格子尺寸

```js
grid.cell = level.box.cellSize;
grid.levelHeight = grid.cell;
grid.width = grid.cols * grid.cell;
grid.depth = grid.rows * grid.cell;
```

`grid.cell` 是箱内唯一真实单位。  
`grid.levelHeight` 也跟随 `grid.cell`，保证高度比例正确。

### 4.3 入箱缩放

```js
function getBoardItemScale() {
  return grid.cell / itemCellSize;
}
```

放入箱内、撤销恢复、提示自动摆放、旋转后回位，都必须使用 `getBoardItemScale()`。

### 4.4 高度计算

箱内 y 坐标必须使用当前入箱缩放：

```js
const boardScale = getBoardItemScale();
item.mesh.position.y = getBoardItemY(item, boardScale, level);
setItemScale(item, boardScale);
```

不能继续用 `scale = 1`，否则小关卡大 cellSize 时，平面尺寸和高度会不一致。

### 4.5 待放区布局

待放区仍然使用：

```js
trayScale = 0.8
traySlotXs = [-1.55, 0, 1.55]
trayZ = 4.85
```

这些值不随关卡变化。

## 5. 当前三关配置

### 第一关：小单入门

```js
box: {
  cols: 2,
  rows: 2,
  levels: 1,
  cellSize: 1.15
}
```

目标：

- 真实 `2x2x1`。
- 通过更大的 `cellSize` 让箱子看起来不小。
- 待放区仍保持统一预览尺寸。

### 第二关：体素叠放

```js
box: {
  cols: 4,
  rows: 4,
  levels: 2,
  cellSize: 0.78
}
```

目标：

- 进入标准尺寸。
- 引入两层高度。
- 让玩家开始理解体素叠放。

### 第三关：叠高挑战

```js
box: {
  cols: 6,
  rows: 6,
  levels: 3,
  cellSize: 0.78
}
```

目标：

- 更大底盘。
- 三层高度。
- 验证固定摄像机和固定待放区约束下的最大可用关卡规模。

## 6. 设计原则

后续新增关卡时遵守以下规则：

1. **逻辑尺寸和视觉尺寸一致**
   - 不做不可放的假空间。
   - 不做视觉箱体大、逻辑网格小的关卡。

2. **小关卡靠增大 `cellSize`，不靠推进相机**
   - 例如 `2x2` 可用 `cellSize: 1.15`。
   - `3x3` 可根据画面测试使用 `cellSize: 0.95` 左右。
   - 标准关建议 `cellSize: 0.78`。

3. **大关卡优先增加规则复杂度，而不是无限扩大底盘**
   - 固定摄像机下，`6x6x3` 已经接近需要重点验证的上限。
   - 如果画面过挤，应减少底盘尺寸，改用高度、异形块、顺序、障碍等方式增加难度。

4. **所有入箱状态必须使用 `getBoardItemScale()`**
   - 手动放置。
   - 提示自动摆放。
   - 撤销恢复。
   - 旋转已放置物品。
   - 错误回位。

5. **待放区永远使用固定预览尺寸**
   - 不受 `box.cellSize` 影响。
   - 这样不同关卡的操作入口稳定。

## 7. 验证清单

每次调整关卡尺寸后验证：

- 第一关 `2x2x1` 箱子看起来足够大。
- 第一关下方待放物品完整显示，没有裁切。
- 第二关 `4x4x2` 箱子位置不漂移。
- 第三关 `6x6x3` 能完整看到箱体和下方 3 个待放物品。
- 拖拽物进入箱内时会从预览尺寸过渡到箱内真实尺寸。
- 放下后物品尺寸和格子完全吻合。
- 物品高度和层高一致。
- 提示自动摆放动画的起点、终点、高度都正确。
- 撤销后物品尺寸正确恢复。
- 重置/下一关不会改变摄像机构图和待放区位置。

## 8. 技术风险

1. **大关卡可能超过固定构图上限**
   - 需要实机验证 `6x6x3` 是否仍然舒适。
   - 若不舒适，不应推进相机，而应回退尺寸或调整玩法结构。

2. **高度缩放必须同步**
   - 如果只缩放 X/Z，不缩放 Y，会出现体素逻辑正确但视觉高度错误。

3. **历史代码中可能仍有 `scale=1` 的箱内高度调用**
   - 后续修改时要检索 `getBoardItemY(item, 1, ...)`。
   - 箱内状态应使用 `boardScale`。

4. **提示动画跨尺寸时要检查缩放插值**
   - `startScale` 来自待放区。
   - `endScale` 必须是 `getBoardItemScale()`。
