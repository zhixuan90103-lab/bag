# Pack 装箱游戏项目文档

日期：2026-07-16  
项目形态：移动端竖屏 Three.js 3D 装箱拼图原型  
目标用途：给后续开发窗口、协作者或新会话快速理解当前项目实现、设计约束和后续方向。

## 1. 项目定位

本项目是一个受装箱、收纳、逻辑拼图启发的移动端 3D 拼图原型。玩家从下方待放区拿起物品，将其摆入纸箱内，完成一个订单后纸箱合上并进入下一关。

当前核心方向不是完全 2D，也不是真实物理模拟，而是：

- 3D 纸箱场景。
- 网格化体素摆放规则。
- 有高度的空间拼图。
- 移动端竖屏优先。
- 强调稳定构图、清晰反馈、短流程关卡。

当前项目仍是原型阶段，但已经包含多关卡、开箱/合箱、提示、撤销、自动摆放、体素支撑检测等核心系统。

## 2. 当前技术栈

- Vite
- Three.js `0.178.0`
- 原生 JavaScript ES Module
- 原生 CSS
- 无 React/Vue 等 UI 框架

主要文件：

- `index.html`：页面入口，提供 `#app` 容器。
- `src/main.js`：核心游戏逻辑、渲染、交互、状态机。
- `src/levels.js`：关卡数据。
- `src/styles.css`：竖屏布局、按钮、结算面板、调参面板样式。
- `docs/IMPLEMENTATION.md`：当前实现细节大文档。
- `docs/research/LEVEL-ADAPTATION-SCHEME.md`：多关卡尺寸适配方案。
- `docs/research/OPEN-BOX-ANIM-IMPL.md`：开箱/合箱动画实现笔记。
- `docs/research/SYNTHESIS-open-close-multi-size.md`：检索综合结论。

常用命令：

```bash

npm run preview
```

## 3. 核心玩法

玩家目标：

1. 查看当前订单。
2. 从下方待放区拿起物品。
3. 将物品拖入纸箱网格。
4. 根据绿色/红色提示判断是否可放。
5. 放满所有物品后完成订单。
6. 纸箱合箱，进入结算或下一关。

当前操作：

- 拖拽物品：`pointerdown / pointermove / pointerup`。
- 旋转物品：右下角旋转按钮，每次绕 Y 轴旋转 90 度。
- 提示：点击“提示”，系统自动演示一个可放物品的摆放过程。
- 撤销：回到上一次手动摆放或提示自动摆放之前。
- 重置：当前关卡重新开始，并重播开箱。
- 下一关：结算面板进入下一关。

当前提示不是“只显示提示”，而是“自动摆放演示”。动画会像手动摆放一样，从待放区拿起、飞到目标位置、落下。

## 4. 关卡数据

关卡定义在 `src/levels.js`。

当前有 3 关：

| 关卡 | 名称 | 尺寸 | cellSize | 目标 |
|------|------|------|----------|------|
| 1 | 小单入门 | `2x2x1` | `1.15` | 小关卡但箱子不显得过小 |
| 2 | 体素叠放 | `4x4x2` | `0.78` | 引入两层高度 |
| 3 | 叠高挑战 | `6x6x3` | `0.78` | 验证大底盘和三层高度 |

关卡结构：

```js
{
  id: 'level-id',
  name: '关卡名',
  box: {
    cols,
    rows,
    levels,
    cellSize
  },
  items: [
    {
      id,
      label,
      shape,
      height,
      color
    }
  ]
}
```

当前 `shape` 已经使用矩形和异形脚印：

```js
rectShape(cols, rows)
shapeCorner()
shapeL3()
shapeL4()
shapeJ4()
shapeT()
shapeU()
```

`height` 默认为 `1`，可配置为 `2` 或 `3`，表示体素高度。

## 4.1 摆放物品生成方式

当前所有摆放物品统一使用 polyomino 圆角挤出管线生成：

```js
createSolidPolyominoGeometry(shape, cellSize, height, inset)
```

这套生成方式同时用于：

- 方形块
- 长条块
- L / J / T / U
- 拐角异形
- 红/绿 ghost 提示平面

核心原则：

- 方形和异形不再分开使用不同几何体。
- 不再让方形走 `RoundedBoxGeometry`、异形走另一套 `ExtrudeGeometry` 参数。
- 所有物品共享相同的圆角、倒角、材质和法线处理口径。
- 逻辑占格仍由二维 `shape` 矩阵决定，圆角只影响视觉，不影响放置判断。

生成流程：

1. 从 `shape` 矩阵提取 polyomino 外轮廓。
2. 轮廓按 `cellSize` 转成局部坐标。
3. 做轻微内缩，保留格子之间的视觉缝隙。
4. 只对外凸角做圆角，内凹角保持稳定，避免斜切伪影。
5. 使用 `ExtrudeGeometry` 挤出高度。
6. 使用小 bevel 做顶面到侧面的圆角过渡。

注意：不使用负 `bevelOffset`。凹多边形配合负 `bevelOffset` 会导致 Three.js 生成错误面，表现为异形块边缘出现斜切。

## 5. 多关卡尺寸适配方案

项目明确约束：

1. **摄像机固定**
   - 不因为小关卡自动推进相机。
   - 不因为不同关卡改变构图中心。
   - 只允许大关卡装不下时拉远。

2. **下方待放区固定**
   - 最多显示 3 个物品。
   - 待放物品预览尺寸、位置、间距不随关卡变化。

3. **箱内视觉和逻辑一致**
   - 不做视觉箱体大、逻辑网格小的假空间。
   - 玩家看到的可用空间必须就是实际可放空间。

当前采用“两套尺寸基准”：

- `grid.cell / box.cellSize`：箱内真实格子尺寸，决定箱体、网格、碰撞、层高、入箱后物品尺寸。
- `TRAY_CELL_SIZE = 0.78`：下方待放区固定预览尺寸，只用于待放区显示和 mesh 原始尺寸。

物品入箱时使用：

```js
boardScale = grid.cell / TRAY_CELL_SIZE
```

因此：

- 第一关 `2x2x1` 可以通过 `cellSize: 1.15` 让箱子更大。
- 下方待放区仍保持统一预览尺寸。
- 放入箱内后的物品和箱内格子真实匹配。

后续关卡设计原则：

- 小关卡靠增大 `cellSize`，不靠推进相机。
- 标准关使用 `cellSize: 0.78`。
- 大关卡优先增加高度、异形块、顺序和规则复杂度，不无限扩大底盘。
- 所有入箱状态必须使用 `getBoardItemScale()`。

## 6. 体素放置规则

当前放置系统是 3D 体素占用，不是简单 2D 高度图。

核心概念：

- `cols`：箱内 X 方向格数。
- `rows`：箱内 Z 方向格数。
- `levels`：高度层数。
- `shape`：物品在 X/Z 平面的占格。
- `height`：物品占用几层高度。

放置检查：

1. 判断 footprint 是否越界。
2. 构建当前体素占用网格。
3. 从底层到最高可放层扫描。
4. 检查目标空间是否为空。
5. 检查完整支撑。
6. 返回第一个合法层级。

完整支撑规则：

- 第 0 层可以直接放。
- 第 1 层及以上，物品 footprint 下方每一格都必须有实体支撑。
- 如果任一格下方为空，则该层不可放。

这保证了当前空间拼图是“有体积、有支撑”的逻辑，而不是只看平面投影。

## 7. 拖拽和反馈

拖拽过程：

1. 点中可见物品。
2. 物品被抬高到 `grid.pickupHeight`。
3. 关闭物品阴影。
4. 根据指针射线和拖拽平面更新位置。
5. 每帧计算候选落点。
6. 显示网格和 ghost 平面。

提示颜色：

- 绿色：可放。
- 红色：不可放。

ghost 目前是平面提示，不是完整 3D 块。这样更清晰，不会遮挡太多箱内物品。

非法放置时使用“意图落脚层”：

- 不再简单把红框画在箱底。
- `baseLevel` 仍然服务于放置规则，表示真实合法层或判定候选层。
- `displayBaseLevel` 专门服务于 UI，控制红/绿 ghost 和网格显示在哪一层。
- 非法时会根据 footprint 覆盖到的可见格子，取下方最高连续堆叠高度作为显示层。
- 如果想放到第二层但缺支撑，红框和网格会出现在第二层表面，而不是沉到底层被已放物品遮住。
- 如果 footprint 部分出界，只统计仍在箱内的格子，尽量让错误反馈保持可见。
- 有高度的物品正常放到底层时，网格仍显示底层落脚格，不因为物品自身高度显示到中间层或顶层。
- 如果物品叠放后会**超过箱子高度**，红框和网格不显示在底层或最高可放 base 层，而是显示在箱子高度上限平面。例如三层箱子会显示第三层顶部网格，让玩家明确理解“超高”而不是“底层碰撞”。

## 8. 提示和撤销

提示功能：

- 扫描当前可见的最多 3 个待放物品。
- 找到第一个合法摆放位置。
- 自动执行摆放动画。
- 动画包含拿起、飞行、落下、缩放。
- 动画期间锁定玩家输入。

撤销功能：

- 使用快照栈。
- 手动成功放置前记录快照。
- 提示自动摆放前记录快照。
- 撤销时恢复：
  - 物品位置
  - 旋转
  - 是否已放置
  - 所在格子
  - 所在层
  - 待放队列
  - 完成状态

撤销不记录“单独旋转已放置物品”的操作，这是当前技术债之一。

## 9. 意图识别（拖拽自动转角）

**模型**：**形状 × 可放格子（红色 Ghost）× 手势（物品快慢）→ Hot / Warm / Cold**（只改 90° 朝向，不代放）。

规则：

- **拿起不转**；**绿不抢**；转后须合法，提交前再验。
- 落点以 **Ghost 脚印是否在棋盘**为准（不是 mesh 中心）；本拖 Ghost 首次进箱才解锁，归位托盘清零。
- **Hot**：邻域或全盘只剩一种其它可放朝向 → 尽快帮转（停稳/慢滑即可）。
- **Warm / Cold**：多可能 → 更稳，多须停稳；进箱冷静/钝化主要作用于此。
- **快慢**：测物品移动速度（格/秒）。正常挪/快甩不转；停稳/慢滑才评估。近端停下应立刻显示停稳。
- UI：顶部拖拽速度提示；侧栏 **Speed 快慢** 调参；`?intentDebug=1` 详细调试。
- 手动点转 **manualLock ~800ms**；帮转提交锁 ~900ms；与提示代放分责。

细节见 `docs/research/INTENT-IMPLEMENTATION-CHANGELOG.md`、`docs/IMPLEMENTATION.md` §19、`SYNTHESIS-intent-sensitivity.md`。

## 10. 开箱和合箱动画

当前纸箱是 RSC 四襟片结构：

- 前 major 襟片
- 后 major 襟片
- 左 minor 襟片
- 右 minor 襟片

阶段机：

```text
boot → opening → play → closing → settle
```

开箱：

1. 整箱从上方落下。
2. Y 轴顺时针 45 度归正。
3. major 襟片外翻。
4. minor 襟片外翻。
5. 前墙和前襟片淡到幽灵透明度。
6. 进入 `play`。

合箱：

1. 前墙和前襟片淡回实体。
2. minor 合拢。
3. major 合拢。
4. 进入 `settle`。
5. 显示结算面板。

开箱/合箱期间：

- gameplay 输入锁定。
- 点击屏幕可跳过到终态。

当前前墙不是完全消失，而是保留 `frontFadeMinAlpha = 0.1` 的幽灵态，便于仍然感知箱体边界。

## 11. 摄像机和画面约束

当前重要约定：

- 摄像机参数固定后不要随意修改。
- 下方待放区位置固定后不要随意修改。
- 多关卡适配优先通过关卡数据处理，不通过相机推进处理。

当前 `fitCameraToBox()` 的策略：

- 默认保持固定构图。
- 小关卡不推进相机。
- 只在大关卡超出安全范围时拉远。
- `target/screen` 保持默认，避免箱子漂移。

这个约束是为了解决之前出现的问题：

- 多关卡切换后箱子位置变化。
- 小关卡相机推进导致下方待放物品被裁切。

## 12. UI 和移动端布局

页面目标是 iOS 竖屏体验。

桌面测试时，`#app` 模拟手机竖屏画布。移动端则占满屏幕。

主要 UI：

- 顶部订单卡片。
- 顶部操作按钮：提示、撤销、重置。
- 右下角旋转按钮。
- Toast。
- 结算面板。
- 横屏提示。
- PC 调参面板：相机、灯光、桌面。

移动端注意事项：

- 使用安全区变量。
- 横屏时显示竖屏提示。
- 调参面板在移动端隐藏。
- 文本和按钮需要避免遮挡 3D 内容。

## 13. 已知技术债

1. `src/main.js` 仍然偏大
   - UI、渲染、规则、输入、动画、关卡切换都在一个文件。
   - 后续建议拆分为 `board`, `items`, `rules`, `input`, `animation`, `levels`, `ui`。

2. 自动化测试缺失
   - 最应该补的是体素规则测试：
     - 越界
     - 重叠
     - 完整支撑
     - 多高度物品
     - 旋转后 footprint

3. 关卡数据还比较简单
   - 当前多为矩形块。
   - 后续需要异形块、障碍、目标层、顺序变化。

4. `6x6x3` 需要实机验证
   - 固定摄像机和固定待放区下，这可能接近当前构图上限。

5. 透明前墙可能有排序风险
   - 当前可用，但复杂模型增加后可能需要单独处理 renderOrder 或 depthTest。

6. 性能仍需移动端压测
   - 圆角几何体和阴影在物品增多后可能有压力。

## 14. 后续开发建议

当用户提出“更新项目文档”“整理项目文档”“反查补漏文档”“把最新实现更新到文档”或“同步到 NotebookLM”时，默认含义包括：

1. 更新本地项目文档。
2. 删除 NotebookLM 中对应项目文档的旧版本。
3. 上传本地最新版本到 NotebookLM。
4. 确认 NotebookLM 源状态为 `ready`。
5. 通过至少一次问答验证 NotebookLM 没有引用旧关卡、旧 MVP 或旧相机适配方案。

如果 NotebookLM 无法访问，必须明确说明本地文档已更新但 NotebookLM 未同步，并记录失败原因。
